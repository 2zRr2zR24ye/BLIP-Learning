# BLIP-Learning
这是一个blip的学习文件夹。

## BLIP和CLIP的核心区别
CLIP：判别式（Discriminative）。它的任务是“匹配”。给它一张图和一个句子，它输出一个相似度分数（0~1）。它只能回答“像不像”，没有能力生成一个描述性的句子。

BLIP：生成式+判别式（Generative + Discriminative）。它的任务是“理解并描述”。给它一张图，它能直接“写”出一段通顺的英文描述（Image Captioning）。它有一张“嘴”（解码器）。

## 解决的两个核心痛点
- 1. 基于编码器的模型不太容易直接转移到文本生成任务，理解和生成对模型的要求是矛盾的。
```bash
  ┌──────────┬────────────────────────────────────┬────────────────────────────────────────┐
  │          │        理解任务 (Retrieval)        │         生成任务 (Captioning)          │
  ├──────────┼────────────────────────────────────┼────────────────────────────────────────┤
  │ 需要     │ 全局特征对齐                       │ 逐 token 的条件生成                    │
  ├──────────┼────────────────────────────────────┼────────────────────────────────────────┤
  │ 关键机制 │ 对比学习（拉近正样本，推开负样本） │ 自回归解码（给定前文预测下一个 token） │
  ├──────────┼────────────────────────────────────┼────────────────────────────────────────┤
  │ 注意力   │ 双向（看到全部上下文）             │ 因果单向（只看到已生成的 token）       │
  ├──────────┼────────────────────────────────────┼────────────────────────────────────────┤
  │ 输出     │ 一个相似度分数                     │ 一个 token 序列                        │
  └──────────┴────────────────────────────────────┴────────────────────────────────────────┘
  ```

- 2. 嘈杂的网络文本对于视觉语言学习来说并不是最优的。

## 提出的两个核心创新点：
- 1. Multimodal mixture of Encoder-Decoder (MED)
- 2. Captioning and Filtering (CapFilt)

## 疑问点1：什么是Cross-Attention
From deepseek v4 pro:
Cross-Attention 是让文本"看见"图像的机制：Q 来自文本（提问方），K 和 V 来自图像（被问方）。文本 token通过注意力权重定位到图像中与自己相关的区域，然后从该区域"拉取"视觉信息来增强自己的表示。在 BLIP 的 MED中，它使得同一个模型既能"理解"图文关系（encoder 模式），也能"根据图像写描述"（decoder 模式）。

# 三种不同模式的对比表
```bash
                  Unimodal Encoder    Img-grounded Encoder   Img-grounded Decoder
                  ════════════════    ════════════════════   ═══════════════════

  Self-Attention  双向 ✓             双向 ✓                 因果（单向）✓
                  与Encoder共享       与Encoder共享          独立参数！

  Cross-Attention 无 ✗              有 ✓                   有 ✓
                                                            与Encoder共享！

  Feed Forward    有 ✓               有 ✓                   有 ✓
                  全部共享            全部共享                全部共享

  特殊 Token      [CLS]              /[Encode/]              /[Decode/]

  输出用途        CLS的表示          /[Encode/]的表示         逐token生成caption

  训练目标        ITC                ITM                    LM
```
*实质上： MED 通过调整 Self-Attention 的模式（双向/因果/不开），同一个模型可以扮演三个不同的角色。*

## 三种模式的对比：
 考虑一个具体的输入：一张图像，描绘了一个小女孩抱着一只橘色小猫，身旁有一道蓝色围栏；以及对应的文本描述 "a little girl
  holding a kitten next to a blue fence." 我们以该样本为例，遍历 MED 的三种功能模式。

  ---
  模式一：Unimodal Encoder — 对比对齐

  第一种模式下，模型表现为双塔编码器架构。视觉 Transformer 将输入图像分割为 patches 并编码为嵌入序列，附加一个 [IMG_CLS]
  token 聚合全局视觉语义。与此同时，文本编码器——标准的 BERT 风格双向自注意力 Transformer——处理输入文本，句首的 [CLS]
  token 与所有文本 token 进行注意力交互，将句子级语义汇聚为单一表示向量。

  此模式不启用交叉注意力。图像与文本两条通路完全独立，分别产出嵌入空间中的两个向量。随后，Image-Text
  Contrastive（ITC）损失最大化正样本对（匹配图文）的余弦相似度，最小化负样本对（batch 内其他图文组合）的相似度。

  该模式面向图文检索任务优化。双向自注意力机制使每个 token 能够获取完整的句子上下文——例如 "holding"
  的表示同时融入了其主语 "girl" 和宾语 "kitten"
  的信息——从而产出高质量的全局文本表示。然而，由于缺少交叉注意力，模型只能在全句粒度上判断图文对应关系，无法进行 token
  级别的细粒度对齐。

  ---
  模式二：Image-grounded Text Encoder — 多模态融合

  第二种模式在文本编码器的每个 Transformer Block
  中，于自注意力层与前馈网络之间插入一个交叉注意力层。文本序列前附加一个任务专用的 /[Encode/] token。

  每个 Block 内的计算分两步进行。首先，双向自注意力处理文本 token，/[Encode/] token
  在此阶段聚合整个句子的上下文语义。关键在于，这一聚合发生在交叉注意力之前，确保 query
  向量在接触视觉特征之前已经携带了充分的语言理解。

  其次，交叉注意力被激活。文本 token（包括 /[Encode/]）作为 query，来自视觉编码器的图像 patch 嵌入作为 key 和
  value。/[Encode/] token 携带着已聚合的句子语义，与所有图像 patch
  计算注意力权重，据此从视觉特征中有选择地汇聚与文本内容相关的信息。

  最后一层 /[Encode/]
  的输出嵌入即为融合后的多模态表示。该向量经过一个线性分类器，输出二分类预测：该文本是否忠实地描述了该图像。

  模式二与模式一的区别是根本性的。模式一基于独立计算的表示产生一个相似度分数；模式二在每一个 Transformer Block
  中显式建模跨模态交互。当例句 "a little girl holding a kitten next to a blue fence"
  进入该模式时，自注意力层首先解析句法语义依赖关系，随后交叉注意力层在图像上对每个语义成分进行定位验证——名词短语 "girl"
  激活了对应儿童空间位置的 patches，"kitten" 激活了动物所在区域的 patches，"blue fence" 激活了背景结构区域的 patches。

  若将例句中的 "kitten" 替换为 "puppy"，"puppy" 的 query 向量将无法在任何图像 patch 中找到匹配的视觉特征，ITM
  分类器会正确判定图文不匹配——这一判别是模式一仅基于全局特征相似度所无法稳定完成的。

  ---
  模式三：Image-grounded Text Decoder — 自回归生成

  第三种模式将双向自注意力替换为因果自注意力，同时保留模式二的交叉注意力层。/[Decode/] token
  标记生成起始，模型以自回归方式逐 token 预测，直至产出终止 token。

  在每一个生成步骤中，因果自注意力将每个 token 的感受野限制在已生成的前缀范围内——模型仅基于当前的部分前缀进行条件预测。
  交叉注意力则以该前缀为指导，在图像特征上进行查询，使模型在预测下一个 token 时能够从视觉内容中获取依据。

  具体而言：给定图像，模型从 /[Decode/] 开始，通过关注全局图像表示预测 "a"。获得前缀 "/[Decode/] a"
  后，自注意力识别出冠词的存在并期待一个名词性中心词；交叉注意力定位到图像中最显著的视觉实体——儿童——从而预测
  "little"。前缀扩展为 "/[Decode/] a little"，此时语法约束要求接续一个与小型实体一致的名词，模型预测 "girl"。在随后的每一
  步中——"holding"、"a"、"kitten"、"next"、"to"、"a"、"blue"、"fence"——模型都在已生成前缀上的自注意力与图像 patches
  上的交叉注意力之间交替，同时维持语言连贯性和视觉忠实度。

  Language Modeling（LM）损失优化每个 ground-truth token 在其前缀与图像条件下的负对数似然。与 VLP 领域此前广泛使用的
  Masked Language Modeling 不同，自回归 LM 直接赋予模型开放式生成能力——它学会了将视觉信息转化为连贯的自然语言序列。

  ---
  参数共享及其设计逻辑

  三种模式共享除自注意力层以外的全部参数，自注意力层在编码器（模式一、二）和解码器（模式三）之间分叉。

  其原因在于架构层面的本质差异。编码器中的双向自注意力使每个 token
  的表示成为完整输入序列的函数——这对于需要全局上下文来消歧的理解类任务是必要的。解码器中的因果自注意力则对未来位置施加掩
  码，强制满足生成任务所需的自回归约束。这两种行为无法在同一组权重矩阵中共存；若强行共享，两者的性能都将受损。

  反之，交叉注意力层在模式二和模式三中承担完全一致的功能：给定源自文本的 query，从图像的 key-value
  对中汇聚相关视觉信息。无论文本 query 来自双向编码器还是因果解码器，交叉注意力的计算逻辑是不变的。同理，token
  嵌入层和前馈网络执行模态无关的通用变换，共享不会产生冲突。

  实验表明，这种选择性共享将参数量从完全解耦的 361M 降低至
  252M，而性能不降反升——双向和因果自注意力各自专业化、互不干扰，共享的交叉注意力层则从多任务训练中获益。论文 Table 3
  中的消融实验证实：除自注意力层外共享所有参数，构成了参数量与性能之间的最优权衡。

  ---
  联合优化

  在单次训练迭代中，视觉 Transformer
  仅执行一次前向传播——这是整个模型中计算开销最大的组件——产出图像特征并缓存。随后，文本 Transformer
  以不同的功能激活执行三次前向传播：

  1. Unimodal encoder（双向 SA，无 CA），计算 ITC 损失。
  2. Image-grounded text encoder（双向 SA，含 CA），计算 ITM 损失。
  3. Image-grounded text decoder（因果 SA，含 CA），计算 LM 损失。

  三个损失求和后联合反向传播。由于大部分参数在三种模式间共享，每次梯度更新同时提升模型在三个维度上的能力——对比对齐、细粒
  度匹配和条件生成——产出一个能够灵活迁移到理解类与生成类下游任务的统一多模态表示。


## CapFilt
视觉-语言预训练需要海量的图文对，但高质量人工标注的成本太高——COCO
  这种数据集只有十几万张图，对于预训练来说远远不够。于是大家转向网络爬取：图片配上网页的 alt-text。问题是，alt-text
  的质量一言难尽。

  拿一张日落湖景照片来说，它可能被配上的文本包括："a flock of birds flying over a lake at
  sunset"——这是为数不多的、确实在描述图像内容的；"from bridge near my
  house"——摄影师标注的拍摄位置，跟图里有什么毫无关系；"stunning sky over walney island, july
  2009"——旅游相册的标题，不是图像描述；"a week spent at our rented beach house"——个人生活记录，图文毫无关联。

  之前的方法，比如 CLIP 和 ALBEF，对这类噪声的态度基本上是"先不管，量大出奇迹"。确实，堆更多的数据量带来了性能提升。但
  BLIP 的作者指出：噪声并没有被规模"消除"，它的负面影响只是被规模红利暂时掩盖了。如果能主动清洗数据、提升信噪比，同等规
  模下模型可以学得更好。CapFilt 就是为这个目标设计的。

  CapFilt 是 Captioning 加 Filtering 的合称，包含两个功能模块：一个负责"写"（Captioner），一个负责"查"（Filter）。整个流
  程需要有一个起点——一个已经在原始数据（噪声 web 数据加少量干净数据）上预训练过的 MED
  模型。这个模型已经具备了一定的视觉理解和文本生成能力，但因为训练数据充满噪声，它的能力上限被锁住了。

  第一步是训练 Captioner。取 MED 中的 Image-grounded Text Decoder，在 COCO
  这个全干净的数据集上做轻量精调，训练目标就是标准的 LM 损失：给定图片，自回归地生成描述文本。精调完成后，让 Captioner
  给 web 数据集中的每一张图片写一句新的描述，称为 synthetic caption。对于一张日落湖景图，它可能生成 "a flock of birds
  flying over a lake at sunset"；对于一张城堡图，它可能生成 "a large stone building with arched windows"。Captioner
  为什么比原始的 alt-text 靠谱？因为它在 COCO
  上学会了"看图说话"的规范：客观描述视觉内容，不写拍摄地点，不加时间戳，不抒发感想。

  第二步是训练 Filter。取 MED 中的 Image-grounded Text Encoder，同样在 COCO 上精调，联合优化 ITC 和 ITM
  损失，目标是学会判断一个图文对是否真正匹配。精调完成后，Filter 要对两类文本逐一审查。首先是原始的 web text：对于每一对
  (I_w, T_w)，Filter 的 cross-attention 层会让文本中的每个词去图像中"找对应"——如果 "birds"
  能在图中定位到鸟群的视觉特征、"sunset" 能匹配到橙黄色的天空区域，那这大概率是一句靠谱的描述。但如果 "bridge"
  在图里怎么都找不到桥梁结构、"house" 也没有对应的视觉区域，Filter 就会判不匹配，把这句 alt-text 丢进垃圾桶。其次是
  Captioner 生成的 synthetic caption：Captioner
  虽然总体更靠谱，但它毕竟是模型，也会有幻觉——比如给一张纯风景照编出来一个不存在的 "dog"。Filter
  同样对这些合成文本做二次把关，不合规的照丢不误。

  第三步是拼凑新的训练数据集。新数据集 D_new 由三部分构成：人工标注数据（COCO、Visual
  Genome）全部保留，这是质量的锚点；原始 web text 中被 Filter 判为匹配的幸存者，它们证明了自己确实是有效的图像描述；以及
  Captioner 生成且被 Filter 放行的合成文本，它们构成了原本数据集中不存在的、新增的训练信号。

  第四步，用这个新数据集从头训练一个全新的 MED 模型。这里有一个容易忽视但至关重要的细节：不是把旧模型在新数据上继续训练
  ，而是随机初始化、完全从头开始。论文的消融实验证明，继续训练旧模型的收益明显不如从头训练——旧模型的参数空间已经被噪声数
  据塑形过了，其中嵌入了由噪声引起的虚假关联，继续训练只是在已被污染的参数空间中做局部修补。用干净数据从头训练，模型才能
  从零开始建立干净的视觉-语言映射。这跟知识蒸馏领域的一个经验法则是一致的：学生模型不应该从教师模型初始化。

  关于 Captioner 和 Filter 的参数共享，论文做了一个关键的消融实验。如果两者共享参数——就像预训练时 text encoder 和 text
  decoder 共享除 SA 层外的参数那样——性能反而会下降。原因是确认偏差：共享参数意味着 Captioner 和 Filter
  拥有同一套视觉-语言对应模式，Captioner 在某个模式上犯的错（比如把灌木丛误判成蓝色汽车并据此生成描述），Filter
  也会犯同样的错，因此无法自我纠正。实验数据也很直白：共享参数时 Filter 报告的噪声比只有 8%，解耦后升至 25%。不是
  Captioner 的解耦版本变差了，而是 Filter 终于不再包庇自己了。

  生成 synthetic caption 的另一个关键选择是解码策略。论文对比了 Beam Search 和 Nucleus Sampling。Beam Search
  是确定性的——每次挑概率最高的词，生成的永远是模型最自信的句子，噪声比低（19%）但输出千篇一律。Nucleus Sampling
  引入了受控的随机性，从累积概率超过 0.9 的词集合中采样，噪声比略高（25%）但表达更加多样。对于一张日落湖景图，Nucleus
  Sampling 可能在某次采样中生成 "birds silhouetted against orange sky over calm
  water"，另一次生成措辞不同但同样合理的变体。这些多样化的表达在被 Filter 筛选后，保留了比 Beam Search
  更丰富的语义信息，让后续训练出的模型学到了更全面的视觉-语言对应关系。

  从更宏观的视角看，CapFilt 可以被理解为一种运作在数据层面而非输出层面的知识蒸馏。传统知识蒸馏让 student 模型在 logits
  空间模仿 teacher 模型的概率分布——信息的丰富程度受限于输出空间的维度。CapFilt 则让 Captioner
  通过自然语言句子来传递知识——一句完整的图像描述远比一个概率向量包含更丰富的语义约束：它不仅告诉模型"这张图里有鸟、有湖
  、有日落"，还隐式地编码了实体之间的空间关系（"flying over"）、视觉属性的表达惯例、以及句法结构本身。Student
  模型不直接模仿 teacher 的输出，而是从 teacher
  策划和清洗过的数据中自行学习。这种间接的、数据层面的知识传递，在视觉-语言预训练这个特定语境下，被证明比直接蒸馏 logits
  更为有效。

  CapFilt 不改变模型架构，不改变训练目标。它只做一件看似简单但效果深远的事：把训练数据变干净。Captioner 负责补写——为每张
  web 图生成一句靠谱的视觉描述；Filter 负责把关——把不靠谱的文本统统拦掉。干净数据配全新模型从头训练，这就是 BLIP 用 14M
  图片能在多项下游任务上超越使用 400M 甚至 1.8B 图片的同类方法的核心原因。
